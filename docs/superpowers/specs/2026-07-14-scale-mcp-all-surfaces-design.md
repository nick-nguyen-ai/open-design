# Scale the MCP to All Surfaces — Design

**Date:** 2026-07-14 · **Status:** approved (brainstormed with user; all recommended options accepted)

## Goal (the /goal gate)

Make every live surface type composable through the skill + MCP pipeline, and prove it with **five fresh samples — one per surface — each at the Fable quality bar**:

1. The MCP exposes five per-surface compose tools (slide-deck, dashboard, project-page, personal-page, technical-explainer) backed by parameterized craft rules and at least one certified live template per surface.
2. An ingestion engine (certifier + scaffolder) makes any newly built world callable without bespoke wiring.
3. The composer skill covers all five surfaces.
4. Five samples, generated by following the skill against real source material, render at `/demo/<slug>` routes, and each passes a ≥2-pass hostile screenshot review at the bank-CEO bar (content AND design).
5. Defects found during sample verification are fixed **skill- or MCP-side** (template slots, descriptor guidance, tool logic, skill checklist), never by hand-editing rendered output — iterating until all five pass.

Batch templatization of the remaining worlds (Phase 2: 17 decks; Phase 3: 9 personal-pages) follows the same machinery but sits AFTER this gate.

## Current state (verified 2026-07-14)

- **33 live (built) worlds** out of 60 catalogue experiences: 20 slide-decks, 10 personal-pages, 1 dashboard (`db-model-monitoring-cockpit`), 1 explainer (`exp-system-architecture`), 1 project-page (`proj-ai-model-validation-hub`). The other 27 manifests are catalogue metadata only — nothing to templatize.
- **3 world-templates** exist (quarter, cutover, tminus — all decks), reachable only through `compose_slide_deck`.
- `CraftRuleId` is a **closed enum** (4 ids) with per-rule hardcoded logic in `apps/mcp-server/src/tools/validate-fill.ts` — it does not scale.
- Registry discovery already auto-ingests `**/*.worldtemplate.manifest.ts` → `packages/registry/generated/world-templates.json`; what is missing is scaffolding + machine certification of new templates.
- The `deck-composer` skill (`.claude/skills/deck-composer/`) is deck-only.

## Approved design decisions

| Decision | Choice |
|---|---|
| Tool shape | **5 per-surface tools** — `compose_slide_deck` unchanged; add `compose_dashboard`, `compose_project_page`, `compose_personal_page`, `compose_explainer`, all thin wrappers over a shared core |
| Descriptor | **v1.1: `slideKinds` → `sections`** — one vocabulary for all surfaces; migrate the 3 existing manifests in one controlled commit |
| Certifier | **Data-driven suite + CLI** — one generic vitest suite walks the compiled catalogue; zero per-world test boilerplate |
| Selection | **Richer targeting metadata** — descriptors gain `briefKeywords`; canonical-brief regression matrix (one brief per templatized world) |
| Fill scope | Compose tools exist for all types but fills only target **live** templates; tools honestly report when no live template fits a brief |

## 1. Parameterized craft rules

Replace the `CraftRuleId` enum in `packages/contracts/src/world-template.ts` with a discriminated union:

```ts
export const CraftRule = z.discriminatedUnion('kind', [
  z.object({
    kind: z.literal('exactly-one'),
    path: z.string().min(1),          // dot-path to an array slot, e.g. 'gates'
    field: z.string().min(1),         // element field to test, e.g. 'status'
    equals: z.string().min(1),        // required value, e.g. 'warning'
    description: z.string().min(1),   // human message echoed in findings
  }),
  z.object({
    kind: z.literal('required-nonempty'),
    path: z.string().min(1),          // dot-path to a string slot, e.g. 'deck.notice'
    description: z.string().min(1),
  }),
]);
```

`validate_fill`'s `checkCraftRules` becomes a generic interpreter: resolve `path`, filter array elements by `field === equals`, require count 1 (`exactly-one`); or require a non-empty trimmed string (`required-nonempty`). Migrate the 3 manifests:

- quarter: `{ kind: 'exactly-one', path: 'kpis', field: 'status', equals: 'off-track', … }` + notice rule
- cutover: `{ kind: 'exactly-one', path: 'nodes', field: 'disposition', equals: 'stays', … }` + notice rule
- tminus: `{ kind: 'exactly-one', path: 'gates', field: 'status', equals: 'warning', … }` + notice rule

New rule kinds are added to the union only when a template needs one (YAGNI). Finding messages carry `description` so fill authors get the same quality of guidance as today.

## 2. Descriptor v1.1: `sections`

`WorldTemplateDescriptor` moves to `schemaVersion: '1.1'`:

- `slideKinds: SlideKindSpec[]` → `sections: SectionSpec[]` (same shape: `kind`, `purpose`, optional `repeats`, `slots`). A deck section = a slide kind; a page section = a page region. `repeats` keeps its narrative-mapping meaning for decks and is simply omitted by most single-page sections.
- `craftRules: CraftRule[]` (from §1).
- New optional `briefKeywords: string[]` — explicit selection targeting authored at templatization time.
- `SlideKindSpec`/`SlideKindRepeats` are renamed (`SectionSpec`/`SectionRepeats`); the fill-side exports (`TMINUS_SLIDE_KINDS` etc.) rename to `*_SECTIONS`.

One controlled migration commit covers: contracts, the 3 manifests + their fill files, registry emit, MCP server tools/tests, gallery lockstep tests, and the deck-composer skill references. The regression trio (demo 43, sample 7, openwiki 7) must stay green through the rename — selection behaviour is untouched.

## 3. Compose core + five tools

Extract the selection/skeleton logic from `compose-slide-deck.ts` into `apps/mcp-server/src/tools/compose-core.ts`:

- filter catalogue templates by `surface`
- score: audience overlap ×2 + intent/brief keyword match (now including descriptor `briefKeywords`) + corporate fit + styleHint hard filter (unchanged weights — behaviour-preserving for decks)
- emit the fill skeleton from the descriptor's `sections`

`compose_slide_deck` delegates to the core with `surface: 'slide-deck'`. Four new tools register with surface-appropriate descriptions (a deck teaches slide kinds + repeats + narrative mapping; a dashboard/page/explainer teaches sections). All five share input schema (context, brief) and the structured error model. When no live template on that surface clears a minimum score, the tool returns a `NO_TEMPLATE_FIT` outcome naming the closest candidates and what the surface's live pool contains — never a force-fit.

## 4. Ingestion engine

**Scaffolder** — `corepack pnpm --filter mcp-server scaffold-template <experience-id>`: reads the live world's directory (its `content.ts` shape and page component name) and generates the three-file skeleton: `<world>-fill.ts` (Zod stub with fields proposed from the content shape + `*_SECTIONS` descriptor stub), `<world>.worldtemplate.manifest.ts`, and the lockstep wiring. Craft judgment — char limits, guidance prose, `briefKeywords`, craft rules — is emitted as clearly marked `AUTHOR:` placeholders that the certifier refuses (so a scaffold cannot ship unfinished). Extraction of the template component itself (Template.tsx from Page.tsx) stays human craft.

**Certifier** — a data-driven vitest suite (`packages/registry/src/certify.test.ts` or equivalent) plus a `certify` CLI wrapper that walks `world-templates.json` and, per template, imports the world's fill module + shipped fill by convention:

1. **Lockstep both directions** — every Zod fill path has a descriptor slot; every descriptor slot resolves in the Zod shape.
2. **Shipped instance parses** — the world's shipped content (exported as a fill, the quarter/cutover/tminus refactor pattern, now mandatory) passes `FillSchema.parse`.
3. **Slot paths resolve** — every descriptor slot path resolves against the shipped instance; limits hold on it.
4. **Craft rules hold** on the shipped instance.
5. **Template-leak scan** — JSX text literals in `*Template.tsx` above a threshold that don't trace to fill data are flagged (allowlist file for chrome labels).
6. **No `AUTHOR:` placeholders** anywhere in the fill module or manifest.

The suite runs in the standard test gate: an uncertified template fails CI, so nothing enters the catalogue unproven. The certifier automates the mechanical checks; the hostile screenshot pass remains the human-quality gate during templatization.

## 5. Selection regression: canonical-brief matrix

A fixture maps one realistic brief per templatized world → the world that must win selection. Grows with every templatization; runs in the MCP server test suite. The existing scripted demos (demo/sample/openwiki) stay as end-to-end stability locks and gain siblings for the four new pilots.

## 6. Skill generalization

`deck-composer` becomes the surface-aware **`experience-composer`** skill (same directory pattern; deck flow preserved verbatim as the slide-deck path):

- Phase 2 (compose) picks the tool by surface; Phase 3 (narrative map) applies to decks and any repeats-bearing sections; single-page surfaces get a lighter "section map" (which sections carry which source facts).
- `fill-authoring.md` and `scaffold-and-verify.md` stay surface-neutral already (slots, limits, honesty rules, template-leak-first checklist) — they gain only surface-specific notes (e.g. dashboards render live-feeling data; the notice rule covers synthetic time-series).
- The skill directory renames to `.claude/skills/experience-composer/`; `.claude/skills/deck-composer/SKILL.md` becomes a one-paragraph pointer stub naming the new skill (existing docs/ledger references stay resolvable).

## 7. Pilots (one per new surface)

The pilots pick themselves — the only live world per surface: `db-model-monitoring-cockpit` (dashboard), `exp-system-architecture` (explainer), `proj-ai-model-validation-hub` (project-page), plus one personal-page chosen at planning time for extraction cleanliness. Each pilot follows the proven pattern: Template.tsx extraction (fill-driven, leak-free), `<world>-fill.ts` (Zod + sections), manifest, shipped-content parity (the live route renders identically from the shipped fill), certifier green, hostile screenshot pass.

## 8. The five-sample goal test

After the engine + pilots ship, run the full skill + MCP pipeline five times — fresh source material per surface, real topics (chosen at execution; each needs a genuine source the way OpenWiki did). For each: intake → compose via the surface's tool → fill authoring → validate loop → scaffold `/demo/<slug>` → build → screenshot every slide/section → ≥2-pass hostile review at the bar. Defects route to their real owner: template slots, descriptor guidance, tool logic, or skill checklist. The goal is met when all five samples pass and all gates (typecheck, lint, unit incl. certifier, build, e2e, regression scripts, brief matrix) are green.

## Phasing

- **Phase 1 (this goal):** §1–§8 — engine, contracts, tools, ingestion, skill, 4 pilots, 5 samples.
- **Phase 2 (after goal):** templatize the 17 remaining deck worlds.
- **Phase 3 (after goal):** templatize the 9 remaining personal-pages.
